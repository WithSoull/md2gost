# Написать отчёт по ГОСТ 7.32

Ты помогаешь пользователю написать технический отчёт в Markdown, который затем будет конвертирован в DOCX через `md2gost convert` (pandoc + шаблон ГОСТ 7.32).

## Шаг 1 — Сбор информации

Перед написанием задай пользователю следующие вопросы **одним сообщением**:

1. **Тема** — полное название отчёта
2. **Автор(ы)** — ФИО, можно несколько через запятую
3. **Дата** — дата документа (формат YYYY-MM-DD)
4. **Организация / учебное заведение** — кто выполнил работу
5. **Разделы основной части** — перечисли разделы, которые нужны (или скажи «стандартные»)
6. **Есть ли рисунки, таблицы, формулы?** — чтобы включить примеры разметки
7. **Нужен ли список литературы?** — если да, попроси перечислить источники
8. **Нужно ли приложение с исходным кодом?** — если да, попроси указать файлы или путь к коду

Дождись ответов перед тем, как писать отчёт.

---

## Шаг 2 — Написать отчёт

Сгенерируй полноценный `.md` файл, строго соблюдая все правила ниже.

### Обязательные правила формата

#### YAML-шапка (всегда первая, между `---`)

```yaml
---
title: "Полное название отчёта"
author: "Иванов И.И."
date: "2026-04-25"
lang: ru
---
```

Поля `title`, `author`, `date` обязательны — без них `md2gost` выдаст предупреждение. Добавь `lang: ru` для корректной расстановки переносов.

#### Иерархия заголовков

В этом проекте нумерация добавляется **автоматически** Lua-фильтром при конвертации. В markdown пиши заголовки без номеров:

```
# Введение        → в DOCX: ВВЕДЕНИЕ (заглавные, без номера, по центру)
## Название        → в DOCX: 1 Название (нумерация, по левому краю)
### Подраздел      → в DOCX: 1.1 Подраздел
## Второй          → в DOCX: 2 Второй
# Заключение      → в DOCX: ЗАКЛЮЧЕНИЕ (заглавные, без номера, по центру)
```

**Правила:**
- `#` (H1) — только для специальных разделов без номера: Введение, Заключение, Реферат, Список источников
- `##` (H2) — разделы основной части (нумеруются автоматически: 1, 2, 3…)
- `###` (H3) — подразделы (нумеруются автоматически: 1.1, 1.2…)
- ❌ Никогда не пропускай уровни: после `#` нельзя сразу `###`

#### Структура по ГОСТ 7.32

```markdown
# Реферат

Краткое содержание работы.

# Введение

Актуальность. Цели и задачи. Объект и предмет исследования.

## Название первого раздела

Текст раздела.

### Название подраздела

Текст подраздела.

## Название второго раздела

Текст раздела.

# Заключение

Выводы по результатам работы.

# Список использованных источников

# Приложение А

Исходный код программы на языке Python.
```

#### Правила размещения медиа

- Параграф со ссылкой на рисунок/таблицу (`@fig:`, `@tbl:`) должен стоять **непосредственно перед** ним — конвертер автоматически привяжет их к одной странице
- Абзацы рядом с медиа не длиннее **~150 слов** (3–5 предложений) — иначе абзац + медиа могут не поместиться на одной странице
- Не размещай два медиа-элемента подряд без текста между ними

#### Рисунки (pandoc-crossref)

```markdown
![Название рисунка](images/schema.png){#fig:schema}

Как видно из @fig:schema, ...
```

> ⚠️ **Не дублируй слово**: `@fig:` уже рендерится в «рис. N», `@tbl:` — в «табл. N».
> Писать «на рисунке @fig:x» → «на рисунке рис. 1» — неправильно.
> Пиши просто «на @fig:x» → «на рис. 1».

- Путь к изображению должен быть **реальным** — md2gost проверит его существование
- `{#fig:name}` — идентификатор, `@fig:name` — ссылка в тексте
- Подпись пишется внутри `![...]`

#### Таблицы

```markdown
: Название таблицы {#tbl:results}

| Параметр | Значение | Единица |
|----------|----------|---------|
| Скорость | 100      | м/с     |
| Время    | 5        | с       |

Результаты приведены в таблице @tbl:results.
```

#### Формулы

```markdown
$$
E = mc^2
$$ {#eq:einstein}

Из формулы @eq:einstein следует...
```

#### Список литературы

```markdown
# Список использованных источников

1. Фамилия И.О. Название книги. — Город: Издательство, Год. — N с.
2. Фамилия И.О. Название статьи // Журнал. — Год. — Т. X, № Y. — С. Z–ZZ.
```

Для ссылок в тексте используй нумерованные ссылки `[1]`.

#### Приложение с исходным кодом

Приложения размещаются после списка литературы. Каждое приложение начинается с заголовка H1 (`# Приложение А`, `# Приложение Б` и т.д.), после которого идёт краткое описание содержимого.

Исходный код оформляется как блок кода, где **первая строка** — подпись листинга в формате `Листинг X.N – Описание`:

````markdown
# Приложение А

Код реализации модуля на языке Python.

```
Листинг А.1 – Реализация основного модуля
import torch
import torch.nn as nn

class Model(nn.Module):
    def __init__(self):
        super().__init__()
        ...
```

```
Листинг А.2 – Функция обучения
def train(model, dataloader):
    ...
```
````

**Правила:**
- Первая строка блока кода **обязательно** начинается с `Листинг` — иначе фильтр не сработает и код выведется как обычный блок
- Нумерация листингов ручная: `А.1`, `А.2` для Приложения А, `Б.1`, `Б.2` для Приложения Б
- Длинные листинги автоматически разбиваются на несколько таблиц (по 45 строк), продолжения подписываются как «Листинг А.1 – Продолжение»
- В DOCX код отображается шрифтом Courier New 10pt внутри таблицы на всю ширину текстовой области

## Шаг 3 — Сохранить и сконвертировать

После написания сохрани файл рядом с исходниками проекта с именем, отражающим тему, например `report.md`.

Затем **сразу запусти конвертацию** (без подтверждения пользователя):

```bash
md2gost convert report.md
```

Если команда завершилась успешно, сообщи путь к полученному `.docx` файлу. Если произошла ошибка — исправь проблему в `.md` и повтори конвертацию.

---

## Памятка по частым ошибкам

| Ошибка | Как исправить |
|--------|---------------|
| Нет YAML-шапки | Добавить `---` блок в начало файла |
| Пропущен уровень заголовка | Добавить промежуточный `##` или `###` |
| Номера в заголовках вручную | Убрать — фильтр добавит их сам |
| Изображение не найдено | Проверить путь, создать папку `images/` рядом с `.md` |
| `@fig:name` без `{#fig:name}` | Добавить `{#fig:name}` к соответствующему рисунку |
| «на рисунке @fig:x» → «на рисунке рис. 1» | Убрать слово перед ссылкой: писать «на @fig:x» |
| «в таблице @tbl:x» → «в таблице табл. 1» | Убрать слово перед ссылкой: писать «в @tbl:x» |
| Кириллица в путях к файлам | Использовать только латиницу и `_` в именах файлов |
| Рисунок/таблица оторвались от текста | Поставить параграф со ссылкой непосредственно перед медиа |
| Абзац + медиа не помещаются на странице | Сократить абзац до ~150 слов |
